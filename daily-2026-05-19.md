## 技術筆記(踩坑紀錄)

### 坑 1:MSIX 應用的檔案虛擬化

Claude Desktop 是 **MSIX 應用**,所有檔案操作經過 Windows MSIX virtualization。

- 表面路徑:`$env:APPDATA\Claude\` ← 這是「虛擬投影」
- 真實路徑:`$env:LOCALAPPDATA\Packages\Claude_pzs8sxrjxfjjc\LocalCache\Roaming\Claude\` ← MSIX 的 LocalCache

**後果:** 當 Claude.exe 跑著時,兩邊看起來同步;但被 `Stop-Process` 強制結束時,virtualization 在 commit 變更期間會有空窗期。檔案可能短暫「消失」、「變空」、或被截斷。

**證據:** 今晚 `claude_desktop_config.json` 原本 1737 bytes,被腳本搬到備份目錄時只剩 148 bytes — 這是 MSIX 競爭條件造成的部分寫入。

**教訓:** 改 MSIX 應用的設定檔不能用「強制結束 + 改檔」的簡單流程。要嘛接受重灌,要嘛在 app 跑著時改(但 app 可能會覆寫回去)。

### 坑 2:Claude Desktop 自動重啟

`Get-Process Claude* | Stop-Process -Force` 殺掉後,**Claude Desktop updater service 會在數秒內自動拉起來**。所以「停掉再改檔」的腳本策略全程不穩。

證據:每次 Stop-Process 後等 5 秒再查,都會看到 10 個 claude 程序又出現。

### 坑 3:PowerShell 必須是「系統管理員」才能改 Hyper-V

- `Enable-WindowsOptionalFeature`、`bcdedit /set` 都需要 admin
- 普通使用者開的 PowerShell 會噴「要求的作業需要提升的權限」、「存取被拒」
- **正確開啟:** 開始選單 → 搜尋 powershell → 右鍵「以系統管理員身分執行」→ 視窗標題列要有「**系統管理員**」字樣才對

### 坑 4:CPU 虛擬化欄位的誤導

`Get-CimInstance Win32_Processor` 查詢:
- `VirtualizationFirmwareEnabled: False`
- `VMMonitorModeExtensions: False`
- `SecondLevelAddressTranslationExtensions: False`

**這三個 False 不代表 BIOS 沒開虛擬化!** 當 Hyper-V 已啟用時,hypervisor 會「吃掉」這些 flag,從 OS 層查反而回 False。

**真正要看的是:**
```powershell
systeminfo.exe | Select-String "Hyper-V"
```
顯示 "A hypervisor has been detected" → 表示虛擬化已啟用、Hyper-V 正在跑

### 坑 5:PowerShell 5.1 的 ConvertFrom-Json case-insensitive bug

PowerShell 5.1 預設處理 JSON key 是 case-insensitive。若 JSON 有 `.c` 跟 `.C` 兩個 key(對 JavaScript 是合法的兩個不同 key),PowerShell 會誤判為「duplicated keys」而報錯。

**結論:PowerShell 5.1 報的 JSON 損壞,不一定真的是壞檔。** 要用 Node.js 或 PowerShell 7+ 驗證。

### 坑 6:Windows Update 動到的範圍很廣

2026-05-14~15 期間安裝的更新:
- KB5087051(2026-05-14)
- KB5092762 Security Update(2026-05-14)
- KB5089549 Security Update(2026-05-15)

**至少動到了** `HypervisorPlatform` Optional Feature(被關掉)。可能還動到了 WebView2 / Edge 元件 / Electron runtime 等,**這可能就是前端 `'replace'` 錯誤的根因之一**(Claude Desktop 5/14 期間也從 1.569 自動更新到 1.7196,跨大版本)。

### 坑 7:我違反「先看證據再動手」原則

中途我推測 `lastSeenRequireCoworkFullVmSandbox: null` 是嫌疑,寫腳本要把它改成空字串。理由只是「欄位名有 Cowork 而且值是 null」,完全沒有 stack trace 證據支撐。使用者制止:「往根源走」。

**錯在:**
1. 沒先取得 stack trace 就跳到修改設定檔
2. 多個失敗的腳本造成 `config.json.bak` 變成 3 bytes 壞檔(差點覆蓋掉好版本)
3. 腳本失敗時 `Write-Host "✅ 已修改"` 仍會印,造成假成功訊號

**教訓:** Diagnostic-driven debugging 要先拿證據(log / stack trace / sentry),不是基於名稱猜測。
