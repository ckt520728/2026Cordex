# 舊程式遷移指南

## 工作流程

1. 先讀目標 module、manifest、lockfile、設定檔及相關測試；不可只依貼上的片段宣稱完整相依圖。
2. 以輸入、主要流程、輸出、副作用、錯誤處理及持久化行為解釋 module。
3. 映射每個函式呼叫的內部相依、外部套件、環境變數、資料結構與呼叫者。
4. 對照目標 dependency/version 的 breaking changes；無可靠 release notes 時標為待驗證。
5. 先建立或指定現況行為測試，再安排最小批次遷移、驗證與 rollback。

## 證據規則

- 對每項相依標示 `已觀察`、`工具解析` 或 `推論`。
- 指定檔案與 symbol；若有行號則一併提供。
- 不把搜尋不到呼叫者解讀成沒有呼叫者。
- 不直接修改程式，除非使用者明確要求實作。

## 輸出格式

### 白話行為說明
### 相依關係圖

使用表格：`來源 symbol | 相依項 | 類型 | 用途 | 證據`；複雜時可加 Mermaid。

### 升級破壞面

使用表格：`風險 | 受影響位置 | 觸發條件 | 嚴重度 | 驗證方式`。

### 遷移計畫

以編號步驟寫出：變更、前置相依、驗證命令/測試、成功標準與 rollback 點。

### 尚待確認

列出缺少的檔案、版本資料與無法從現有證據判斷的事項。

## 完成前檢查

- 每個計畫步驟都具可執行驗證。
- 相依圖同時涵蓋 outbound dependencies 與主要 callers。
- 風險描述對應到實際程式位置或明確標記為推論。
